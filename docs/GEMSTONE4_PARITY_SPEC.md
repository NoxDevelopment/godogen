# GemStone IV Parity Spec — the text-MUD/MMORPG build bible

> Reference target for the `text-rpg-mud` flagship (Godot client + Evennia BSD server). Compiled from web research (Play.net, GSWiki/Elanthipedia, fan client docs) by workflow wf_91a31fc8. Parity = GemStone IV's UI/UX, systems depth, world + lore, and living-world autonomous agents, built with our own assets/apps. Nothing is 'done' until Jesus vets it.

**Architecture keystone:** GS4's client is essentially a *tag renderer* over one server-pushed markup stream. We adopt that: define the tag/style protocol as our wire format, make the Evennia server the emitter, and the Godot client a stateless renderer (docked panels + point-and-click verb menus + themeable presets). Our WORLDBUILDER feeds rooms/entities/lore, the COMPANION library drives autonomous NPC personalities (consume-only), LLM generation writes descriptions/dialogue, and the RULESET builder defines the numbers the vitals bars render.

**Status:** Evennia server LIVE on Postgres (telnet 4000 / websocket 4002 / webclient 4001); living-world mechanic PROVEN (autonomous world-clock + wandering agent, verified uncached). This spec drives the depth build from here.

## GemStone IV client UI/UX parity spec — front-end layout, text-stream tag taxonomy, and Godot panel plan (Wizard / StormFront-Wrayth / Genie + Lich)

GemStone IV is played through three front-ends that all consume the SAME server-pushed markup stream, so the client is essentially an XML-tag renderer. The server emits a hybrid of plain game text and inline control tags; the front-end routes tagged spans to panels (vitals bars, hands, compass, injuries figure, active-spell timers, side windows) and applies style presets (roomName, speech, whisper, thought, monsterbold, familiar, voln) to color the main scroll. Simutronics' WIZARD (classic, 2000) is a floating multi-window layout with a compass rose, hands line, H/M/S body graphic, roundtime clock, mana-pulse timer and 5 quick buttons. STORMFRONT/WRAYTH (2003, the rich GUI) formalized this into a docked three-pane layout driven by ~60 XML tags (streamWindow/pushStream, dialogData/progressBar, compass/dir, indicator, left/right, spell, a/d clickable links) with point-and-click context menus, an injuries body diagram, and an Active Spells dialog with per-spell countdown meters. GENIE is the fan client (user-defined #window panels, highlights, auto-map, runs Wizard/Wrayth scripts); LICH is a Ruby proxy that sits between server and any FE. To hit GS4 parity in Godot we replicate the tag taxonomy as our server protocol and build a docked panel client that renders it: a central RichTextLabel scroll + a status strip (4 vitals bars, RT bar, stance/encumbrance/mind-state, posture icons), hands line, compass rose, and dockable side panels (room, inventory, active spells, injuries figure, thoughts/ESP, logons, death, group, familiar). Every span in the main stream is a semantic style class and every interactive noun is a clickable link carrying an object id + verb menu.

### Findings

**Three official/fan front-ends — what each is and which to model**  
WIZARD (WizardFE, ~2000): Simutronics' classic Windows client. Floating/resizable multi-window interface; game text can be routed to separate windows (Thoughts, Death, Familiar, Arrivals, Wounds, Pictures). Shows a Hands line (left/right + prepared spell), a compass/Direction display (compass rose, yellow arrowheads = available exits), a Health display (Health/Mana/Spirit bars + small body graphic), a Status display (field-exp saturation, defensive stance, active spell effects), a Roundtime display (current + spell RT), a 2-minute Mana Pulse timer, a bottom command buffer, and 5 quick-action buttons (equip/unequip weapon+shield). Macro tokens: \r (Enter), @ (cursor), \? (prompt for input), \p (pause). STORMFRONT / WRAYTH (2003, renamed Wrayth ~2020): the rich point-and-click GUI, the primary parity target. Docked panes, click any text to get a verb context-menu, XML-driven dialog panels, skins (Original Storm, StealthFront, NishiFront, Brute), per-window logging (floppy icon), toggle all panels with the {panels} command. GENIE (Genie3, genieclient.com): third-party Windows client for all Simutronics games; user-creatable windows via #window, highlights/aliases/variables/scripting, auto-mapping, runs Wizard/Wrayth scripts. LICH: Ruby scripting engine that runs as a PROXY between the server and whatever FE you use (Wizard/Wrayth/Genie) — not a renderer, it injects/reads the same tag stream. Also: Profanity (terminal FE), Avalon (Mac), Web Client, Java client all consume the same protocol.
  
_Build note:_ Model the Godot client on StormFront/Wrayth (docked panes + point-and-click + tag-driven panels). Keep Wizard's floating-window option as a 'classic mode'. Treat the server->client markup as OUR wire protocol so scripting layers (our Lich-equivalent) can proxy it.

**The whole client is a tag renderer — protocol is the architecture**  
All FEs consume one server-pushed stream that interleaves plain game text with inline SGML/XML-ish control tags. The protocol page documents 60+ elements. Categories: system (mode, playerID, settingsInfo, app, endSetup), stream mgmt (streamWindow, pushStream/popStream, clearStream, stream, exposeStream), dialog/UI (openDialog/dialogData/closeDialog/exposeDialog, progressBar, cmdButton, label, image, closeButton), character state (prompt, roundTime, castTime, indicator, spell, left, right), navigation (a, d, compass, dir, nav), components/containers (compDef, component, container, clearContainer, exposeContainer, deleteContainer, inv), text formatting (pushBold/popBold, b, output, style, preset), settings/control (flag, menu, mi, monopolize, switchQuickBar, updateverbs), plus skin, dropDownBox, upDownEditBox, menuLink, radio, link, sep, pushInputState/popInputState, cmdlist, cmdtimestamp. mode id values: GAME/LOGIN/CMGR.
  
_Build note:_ Define this tag set as our canonical text-protocol schema first; the Godot client is a stateless renderer over it and the Evennia server is the emitter. This is the single most load-bearing decision for GS4 parity.

**Main text stream — styling / preset taxonomy**  
The central scroll interleaves everything not routed elsewhere. Styling is driven by <style id=...> and <preset id=...> spans plus bold (<pushBold>/<popBold> or <b>) and font switches (<output class='mono'> for fixed-width tables, empty class to return to proportional). Named presets/style ids seen: roomName (room title line), roomDesc, speech (player says/asks/exclaims), whisper, thought / thoughts (ESP), monsterbold (creature names — the bold-highlighted nouns you can attack/target), familiar (familiar-relayed text), voln (society channel), plus the highlight system (preset highlights + user custom highlights + a color palette accessed by right-click > Settings). Each preset maps to a user-editable fg/bg color and font. monsterbold is special: creature/notable nouns are wrapped so they render bold AND are clickable targets.
  
_Build note:_ In Godot use a RichTextLabel with BBCode; map each preset id to a theme color/style. Ship sensible defaults (room title = tan/gold, speech = pale green, whisper = dim, thought/ESP = cyan/blue, monsterbold = bright bold, familiar = purple, self-actions = white) and make every preset a user-remappable palette entry. Support a monospace toggle for aligned tables (EXP, INVENTORY, SKILL).

**Clickable text + point-and-click verb menus**  
Interactivity is inline: <a exist='OBJECTID' noun='NOUN'>display text</a> marks any object (item, creature, player, exit) as clickable and carries a persistent object id + its noun; <d cmd='COMMAND'>text</d> is a direct command link that runs COMMAND on click. In StormFront, left/right-clicking a highlighted noun opens a context menu of applicable verbs (LOOK, GET, ATTACK, ANALYZE, etc.); the verb list is refreshed by <updateverbs>/<menu>/<mi coord=...>. Links also carry char/game/coord attributes for cross-game and coordinate menus. <link URL=...> and launchURL open external browser pages.
  
_Build note:_ Emit <a exist noun> for every referenceable entity from the server so the Godot client can make nouns clickable. Implement a right-click radial/context menu that requests the verb list for that object id, then sends the chosen verb+noun as a command. This is core GS4 UX and where our LLM/worldbuilder object ids plug in.

**Vitals / status bars (Health, Mana, Spirit, Stamina) + derived meters**  
Driven by <progressBar id=... value=0-100 text='...' top left height width align tooltip>. Documented progressBar ids: health, health2 (secondary/overlay), mana, spirit, stamina, pbarStance (stance meter), encumlevel (encumbrance), mindState (experience saturation/mental acuity), nextLvlPB (progress to next level). Each bar shows a fill value 0-100 plus a text overlay (e.g. 'health 100/100', 'mana 84/120'). Colors are user-customizable (defaults roughly: health red/green, mana blue, spirit yellow/gold, stamina green). Health/Mana/Spirit/Stamina come from the HEALTH/MANA/etc. verbs and update live.
  
_Build note:_ Build a StatusStrip with four labeled ProgressBar widgets (health, mana, spirit, stamina) each with a value + text overlay, driven directly by progressBar tags keyed on id. Add pbarStance, encumlevel, mindState, nextLvlPB as additional bars. Keep colors themeable per bar (StormFront lets users recolor each).

**Roundtime (RT) and cast time bars**  
<roundTime value='SECONDS'> sets an action lockout; <castTime value='SECONDS'> is the spell-prep timer (shown separately in Wizard's Roundtime display which shows current RT AND spell RT). The server also sends <prompt time='UNIXTIME'> before each command prompt, so the client knows current server time and can count RT down accurately. Classic display is a shrinking colored bar / numeric countdown; combat text prints 'Roundtime: 5 sec.' and the bar drains over those seconds. During RT typed commands queue or are rejected.
  
_Build note:_ Render an RT bar that fills to roundTime.value and drains to zero using prompt.time as the server clock reference (compute remaining = (rt_start + value) - server_now). Show a separate cast-time bar for castTime. Gray/queue the input line while RT>0 as an option.

**Stance, encumbrance, and posture/status indicator icons**  
Defensive stance shown via pbarStance progressBar and in the Status display (offensive<->defensive). Encumbrance via encumlevel progressBar (none/light/... /overloaded affecting RT and dodge). Posture & status conditions are boolean icons via <indicator id=... visible='y|n'>. Documented indicator ids: IconKNEELING, IconPRONE, IconSITTING, IconSTANDING, IconSTUNNED, IconHIDDEN, IconINVISIBLE, IconDEAD, IconWEBBED, IconJOINED. Only one posture (standing/sitting/kneeling/prone) is active; stunned/hidden/invisible/dead/webbed/joined layer on top.
  
_Build note:_ Add a row of toggleable status icons bound to indicator ids (posture group = mutually exclusive; condition group = independent flags). Bind stance + encumbrance to their progressBars. joined = grouped/following indicator.

**Hands display (left / right / prepared spell)**  
<left exist='OBJECTID' noun='NOUN'>item text</left> and <right ...>item text</right> report what is held in each hand (empty when exist absent). <spell>SPELL NAME</spell> reports the currently prepared/readied spell (e.g. 'Minor Shock', or 'None'). Classic Wizard shows this as a three-cell line: Left Hand: ... | Right Hand: ... | Spell: ....
  
_Build note:_ Build a HandsPanel with three fields (Left Hand, Right Hand, Prepared Spell) bound to left/right/spell tags; items are clickable (carry exist/noun) to open the same verb menu. This is glanceable combat state — keep it near the input line.

**Compass rose + directional movement + obvious exits**  
<compass> contains <dir value='...'> child elements for each currently available exit; value set = n, s, e, w, ne, nw, se, sw, up, down, out. The FE renders a compass rose (8 compass points + up/down/out) and highlights only the directions present (Wizard: yellow arrowheads on available exits). Clicking a lit direction sends the movement command. Separately, the room text prints a verbatim 'Obvious exits:' (or 'Obvious paths:') line, e.g. 'Obvious exits: east, northwest'. <nav rm='ROOMID'> announces room changes for auto-mappers.
  
_Build note:_ Build a CompassRose widget (8 diagonal/cardinal buttons + Up/Down/Out) driven by the compass/dir set; lit buttons are clickable and send movement. Also parse/emit the textual 'Obvious exits:' line in the room stream. Emit nav rm=... on every room transition for our auto-map.

**Room display format + room components/window**  
Room prints as: a bracketed title line [Area, Room Name] styled roomName; then the description paragraph (roomDesc); then a 'You also see ...' objects line; then 'Also here:' listing players/creatures/bodies present (e.g. 'Also here: Yaristen, Loralai, the body of Lord Reminion who is lying down'); then the 'Obvious exits:'/'Obvious paths:' line. StormFront also mirrors current room into a dedicated Room window via component tags: <component id='room desc'>, <component id='room objs'> (the 'you also see' objects), <component id='room players'> (the also-here list), <component id='room exits'>, and <component id='sprite'> (room art id). Components are re-pushed on any change so the Room window stays live.
  
_Build note:_ Render the room in the main scroll AND mirror into a dockable Room panel using the 4 components (desc / objs / players / exits). Title line clickable-styled, objects and players carry <a exist noun> for verb menus. sprite id -> optional room illustration (feed from worldbuilder art). This is where the worldbuilder's room/entity data maps 1:1.

**Active Spells window with per-spell countdown timers**  
The Active Spells dialog (StormFront) graphically lists every active spell/effect with a timer and a meter showing remaining duration; users right-click an individual spell to recolor its bar. Populated by dialogData/progressBar tags in a dedicated dialog. The SPELL ACTIVE verb lists actual durations sorted: miscellaneous effects first (cooldowns, combat-maneuver cooldowns), then positive effects, then negative effects, then by spell number. Combat maneuver cooldowns also appear here.
  
_Build note:_ Build an ActiveSpells dock: one row per effect = name + draining ProgressBar (duration) + numeric time left, fed by dialogData targeting an 'activespells' dialog with a progressBar per effect. Sort misc/cooldowns, buffs, debuffs, by id. Per-row recolor. Our ruleset builder defines spell durations; the server emits the countdowns.

**Injuries / wounds / scars — body-part diagram**  
14 wound locations: head, neck, chest, abdomen, back, right eye, left eye, right arm, left arm, right hand, left hand, right leg, left leg, nervous system (nerves). Each location can carry a WOUND and/or a SCAR at severity rank 1/2/3. Rank1 = minor (combat-maneuver penalties only; rank1 scars have NO penalty); Rank2 = moderate (usually bleeds, substantial penalties, can restrict actions); Rank3 = severe (restricts activities per location — e.g. rank3 right hand can't hold/swing a weapon or cast most spells; head wounds block searching/spellcasting). Scars = partially-healed wounds (herbs leave scars; empaths heal without scarring). Wizard has a graphical Wounds window (body representation); StormFront shows an injuries figure/panel where each body part is color-coded by worst wound/scar severity. HEALTH verb prints wounds+scars, max/current HP, spirit, stamina (+recovery), phys/magic damage reduction, bleeding location+rate/round, poisons, diseases, overexertion.
  
_Build note:_ Build an Injuries panel: a front/back humanoid figure with 13 hit-region overlays + a nerves indicator, each tinted by worst wound(red)/scar(blue?) at rank 1/2/3 (light->dark). Drive it via indicator/dialogData tags per location (e.g. injury id + severity). Also parse the HEALTH text block. Our ruleset builder owns crit->wound tables; server emits per-location state.

**Experience / mind-state ('as clear as a bell' ladder)**  
Field-experience pool fills as you act; the mental-acuity ladder (from the EXP verb, empty->full, as fraction of capacity): 'clear as a bell' (0.0), 'fresh and clear' (>0.0), 'clear' (0.25), 'muddled' (0.5), 'becoming numbed' (0.62), 'numbed' (0.75), 'must rest' (0.90), 'completely saturated' (>1.0, imperative to rest). Pool capacity ~= 800 + LOG + DIS + Wisdom bonus + subscription bonus. Experience absorbs from the field pool into permanent XP via periodic 'pulses' (on-node ~25/pulse, in-town ~22, elsewhere ~19; modified by Logic, pool fullness, group). mindState progressBar reflects saturation; nextLvlPB reflects progress toward next level.
  
_Build note:_ Bind mindState progressBar to the saturation fraction and label it with the current ladder string; bind nextLvlPB to level progress. Show a mana-pulse / absorption-pulse timer (Wizard had a 2-min mana clock). Our ruleset builder can redefine the ladder labels/thresholds per game.

**Side / stream windows and how output is routed to them**  
<streamWindow id=... title=... subtitle=... location='left|right|center' target=... resident=... save=... scroll=... timestamp=...> defines a dockable window; <pushStream id='X'>...text...<popStream/> routes the enclosed text to window X instead of main; clearStream/exposeStream manage them. Documented stream ids: main, room, inv (inventory), Spells, familiar, thoughts (ESP/LNet), logons (arrivals/logins), death (death notices), speech, ambients, announcements, bounty, society, loot, charprofile, charsheet. StormFront's canonical panes: center = main story; a left long box = Spell list; a right long box = Backpack/inventory contents; top-left = experience bar; a Combat window with an adjacent Injuries/scars/health window. Thoughts window: with it open, ALL ESP messages go there instead of the game window. ESP system (THINK/CHAT/ESP TUNE, IC+OOC channels, 2020 overhaul) + fan LNet chat both feed the thoughts stream; scripts can reroute private channels to the familiar window. Containers use <container id=... title=... location=...> + <inv id=...> lines + clearContainer/exposeContainer/deleteContainer for open backpacks/boxes.
  
_Build note:_ Build a docking system where each stream id = a TabbedPanel/dock target; pushStream/popStream route spans. Ship default docks: Room, Inventory, Active Spells, Injuries, Thoughts/ESP, Logons, Death, Group, Familiar, Bounty/Society, Loot. Container tags open live per-container content panels (nested backpack view). Let users add custom windows (Genie #window parity).

**Command input, history, quick bars, and input state**  
Bottom command line sends raw commands; <prompt time=...> marks each prompt (also the RT clock source). Command history recall (up/down). Quick-action buttons: Wizard's 5 buttons; StormFront <cmdButton id value cmd echo tooltip>, <switchQuickBar id>, and macro/hotkey bars are server- or user-defined. <pushInputState state=...>/<popInputState> switch input mode (e.g. password masking at login, menu selection, CMGR character-manager). <monopolize id='main'> temporarily blocks other streams (modal prompts). <cmdlist>/<updateverbs default=...> supply the default verb menus.
  
_Build note:_ LineEdit input with history stack + tab-complete on visible nouns; a configurable quick-button/macro bar bound to cmdButton tags; honor pushInputState for password masking and modal menus; support a login/character-manager mode (mode id=LOGIN/CMGR). Support macro tokens (Enter, cursor pos, prompt-for-input, pause) for Wizard-parity scripting.

**Concrete Godot client layout (widget tree)**  
Recommended docked layout mirroring StormFront: TOP BAR = experience/level (nextLvlPB) + mind-state label + game/char title (app tag). LEFT COLUMN (VBox, dockable) = Active Spells list (top), then a general stream tab group. CENTER (largest) = main story RichTextLabel scroll (BBCode). RIGHT COLUMN = Room panel (desc/objs/players/exits) over Inventory/backpack container view. BOTTOM STATUS STRIP (above input) = [4 vitals bars H/M/Sp/St] + [RT bar + cast bar] + [stance + encumbrance + mind-state bars] + [posture/status icon row] + [Hands line L/R/Spell] + [Compass rose]. BOTTOM = command LineEdit + quick-button bar. FLOATING/TABBED (toggle with a {panels}-style hotkey) = Thoughts/ESP, Logons, Death, Familiar, Group, Bounty, Society, Loot, character sheet/profile. All panels dockable/detachable/resizable with saved layouts + skins/themes.
  
_Build note:_ Implement with Godot Control nodes: root = a docking container (or use a plugin like Godot's built-in split containers / a docking addon). Central = RichTextLabel (scroll_following, BBCode on, monospace toggle). Bars = ProgressBar keyed by tag id in a dictionary so the renderer just does bars[id].value = v. Persist layout + palette to user settings (StormFront exports/imports XML layout — offer JSON layout export/import).

### How our apps feed it

- WORLDBUILDER -> feeds the room/entity data behind the tag stream: room titles (roomName), descriptions (component 'room desc'), objects ('room objs'), exits (compass/dir + component 'room exits'), room art ids (sprite), and the persistent object ids used in <a exist=...> clickable nouns. Timelines/lore drive ambient/announcement streams.
- COMPANION AI library (consume-only) -> drives autonomous NPCs whose speech (preset speech), whispers, ESP thoughts (thoughts stream) and emotes flow through the same styled main stream and 'Also here:' list; their personality/psychology decides verb choices behind clickable NPC menus.
- LLM GENERATION -> produces room descriptions (roomDesc), NPC dialogue lines (speech/whisper), thought/ESP chatter (thoughts), and combat/ambient flavor, all emitted already wrapped in the correct style presets + clickable <a exist noun> spans.
- RULESET BUILDER -> defines the numbers the bars render: HP/mana/spirit/stamina maxima (progressBar values), roundTime/castTime durations, the wound/scar location+rank crit tables (injuries figure), spell durations (Active Spells countdowns), the experience mind-state ladder labels/thresholds (mindState/nextLvlPB), stance and encumbrance effects.
- GODOT CLIENT (this build) -> a stateless tag renderer over our text-protocol: docked panels + point-and-click verb menus + themeable style presets + saved layouts; classic 'Wizard mode' (floating windows) and rich 'StormFront mode' (docked) as two presets.
- LICH-EQUIVALENT scripting proxy (future) -> our automation layer sits between Evennia server and the Godot client reading/injecting the same tag stream, so scripts work without touching the renderer.

## GemStone IV — World, Lore & Living-World / Autonomous-Agent parity spec (for the text-MUD/MMORPG template: Godot client + Evennia server)

GemStone IV (Simutronics, set in the world Elanthia / continent Elanith, in-fiction "Modern Era" year = real year + 20, so mid-2026 ≈ 5126 M.E.) is a 35-year-old text MMORPG whose depth comes from four pillars we can mirror: (1) a large hand-authored WORLD of interconnected towns each with its own culture and a ring of level-tiered hunting areas; (2) deep LORE — a 30+ member Arkati pantheon split into Liabo (light), Lornon (dark, incl. the destruction "trinity" Luukos/Sheru/Marlu), and Neutral, plus 13 playable races, the eight Elven Great Houses, and a 100,000-year timeline (Drakes → Ur-Daemon War → Ages of Darkness/Chaos → Elven Empire → Turamzzyrian Empire → Modern Era storylines); (3) tick-driven CREATURE ECOLOGY (spawn/despawn tied to player presence, population caps, wandering, aggro, AS/CS/DS/TD combat, maneuvers, treasure, skinning, respawn); and (4) a LIVING WORLD driven by GameMaster storylines/festivals, a strong player-RP culture, a 1:1 real-time Elanthian calendar whose months/days are named after the gods, regional weather, a silver economy with keeper-run shops (furrier/gemshop/pawnshop/bank) and player-run shops, a dynamic Adventurer's Guild BOUNTY system (10 procedurally-assigned task types), and three anti-undead/anti-Grimswarm Societies. We can reach parity — and exceed GS4 because GS4's "life" is largely GM-driven and manual — by simulating autonomous agents with Evennia tick loops (world clock, spawners, NPC daily schedules, faction pressure) enriched by our Companion-AI personalities (consume-only) and LLM-generated flavor. This spec ends with a concrete first region: a Wehnimer's-Landing-style starter harbor town (~30 rooms, keeper-run shops, scheduled townsfolk agents) plus a tiered newbie hunting area (sewer rats → environs kobolds/goblins → old-road hobgoblins/gnolls).

### Findings

**Continental geography of Elanith (the parity world map)**  
Elanthia is the world; Elanith is the main known continent (part of the greater landmass Jontara, north of the continent Finnia). Regions/towns: DARKSTONE BAY region — Wehnimer's Landing, the oldest/original human frontier hub. NORTHWATCH (far north) — Icemule Trace (halfling town) and Pinefar Trading Post. ELVEN NATIONS (east, across the DragonSpine mountains via the Locksmehr Trail) — Ta'Illistim (scholar-city capital), Ta'Vaalor (fortified warrior city-state), Cysaegir, and the hamlet Sylvarraend. TURAMZZYRIAN EMPIRE lands (south/southwest) — Solhaven/Vornavis (free port), River's Rest (bayou town). ISLES — Kharam-Dzu, the dwarven mining town on Teras Isle; Kraken's Fall. DWARVEN — Zul Logoth, an underground city on the Locksmehr Trail through the DragonSpine. Key features: DragonSpine mountain range (major east-west barrier), Darkstone Bay, Lysierian Hills, Dragonsclaw Mountains. **Creature (mob) mechanics — the ecology engine**  
SPAWN: creatures spawn when adventurers are detected in a hunting area and become active. WANDER: active creatures roam the area looking for targets; rarely they wander into adjacent zones or even into a town. DESPAWN: when adventurers leave, active creatures have a chance to despawn, and any items they carried are removed from the world. COMBAT: each creature has level, family, body type, undead flag; attacks use an Attack Strength (AS) that varies by stance and health; some cast warding spells at a fixed Casting Strength (CS); many use combat/creature MANEUVERS (special moves). Defenses: Defensive Strength (DS), Unarmed Defense Factor (UDF), Target Defense (TD). An ACTION TIMER governs how often a creature acts (≥12s = slow, ≤6s = fast; haste halves it). LOOT: on death, SEARCH/loot yields treasure (silvers, gems, boxes, items); some creatures are SKINNABLE (pelt value depends on First Aid + Survival skill, weapon used, kneeling, dexterity, and whether the skin area was damaged). Aggressive creatures attack on sight; others are passive until provoked.
  
_Build note:_ Implement as Evennia scripts: (a) a per-zone Spawner (ticker) that keeps population between min/cap only while players are present, tracks respawn timers, and cleans up loot on despawn; (b) a Creature typeclass with AS/CS/DS/TD/action-timer stats + a maneuver list + a loot table + optional skin entry; (c) a Wander behavior (random-walk within zone bounds, small leak chance to adjacent room); (d) Aggro on player entry for aggressive families. Define all stat blocks and loot/skin tables in the RULESET BUILDER.

**SYNTHESIS — how WE simulate autonomous agent lives in text (tick loops + Companion AI + LLM)**  
Layered simulation on the Evennia server: (A) WorldClock ticker derives Elanthian date/time (1:1 real time) → drives day/night, seasons, holidays, and every schedule. (B) Zone Spawners run creature ecology (spawn only near players, population caps, respawn timers, wander, aggro, loot/skin, despawn-cleanup) — no LLM in the hot loop, pure rules for determinism/cost. (C) NPC Schedule state-machines: each townsfolk agent has a daily routine keyed to the clock (baker bakes at dawn, tavern fills at night, guards patrol a beat, a wandering merchant circuits towns weekly, children play in the square by day). (D) Companion-AI binding (consume-only): named NPCs each instantiate a Companion personality (FFM/HEXACO traits, cultural origin matched to race, emotional state, episodic memory of the player) that selects/colors dialogue and reactions — so the furrier remembers you, the crier's mood shifts with events. (E) LLM flavor layer (cached/budgeted): authoring-time room descriptions, on-demand ambient one-liners, crier announcements tied to real world-state, weather prose, bounty/quest flavor, and event beats — always constrained by lore-graph context to stay canon. (F) Faction pressure meters (Luukos undead, Grimswarm, etc.) tick toward thresholds that fire EventDirector incursions/festivals, giving emergent, GM-free 'sagas.' Net effect: a world that ages, works, worships, trades, and fights on a clock — matching GS4's hand-run liveliness but automated and always-on.

## GemStone IV systems depth — parity spec + prioritized build checklist for the Godot/Evennia text-MMORPG template

GemStone IV (Simutronics, set in Elanthia) is a d100 open-ended, roundtime-gated MUD whose depth comes from a small set of tightly-interlocking, fully numeric systems — all of which are natural fits for our data-driven RULESET builder + computed engines, an authoritative Evennia (BSD) server, a Godot text client, and a WORLDBUILDER/companion-AI/LLM content pipeline for the living world. The load-bearing loop is: 10 stats -> stat bonuses -> AS/DS/CS + training points -> skills -> combat/magic resolution (AS+AvD+d100-DS>100; damage=(endroll-100)xDF; critical rank from armor divisor) -> wounds/HP/bleed -> healing -> experience into a field pool -> mind-state-gated absorption -> levels/training. Everything below is a table + a pure function, which is exactly what a ruleset builder authors and a computed engine evaluates; the server enforces roundtime + pulses + persistence, the worldbuilder authors Elanthia (rooms, shops, nodes, society HQs, spawns), companion-AI gives NPCs inner lives, and LLM generates room/critical/dialogue text. Primary sources (all authoritative): GSWiki Combat_system, Statistic, Character_creation, Skill, Training_point, Magic, Spell_circle, Critical, Wound, Experience, Spellburst, Combat_Maneuvers, Enchant, Silver_coin/Bank/Locker/Box/Trap, Order_of_Voln/Guardians_of_Sunfist/Society (gswiki.play.net). Numbers below are quoted from those pages. Build the vertical slice in the checklist order; treat every table as ruleset data so a new authored world can reskin GS4 mechanics without code changes.

### Findings

**Professions (10 + Savant) and the square/semi/pure tier system**  
Ten playable professions grouped by spellcasting tier. PURE (3 spell circles, physically weak, cannot wear heavy armor, expected to train Spell Research ~2x/level, and each gets Alchemy via guild skills): Wizard, Cleric, Empath, Sorcerer. SEMI (1 minor + 1 professional/base circle, hunt primarily with a weapon supported by magic, wear >= hard leather, train weapons ~2x AND Spell Research 2x): Bard, Ranger, Paladin. SQUARE (up to 2 minor circles, combat-focused, heavy armor, little/no spell research): Warrior, Rogue, Monk. Hybrids: Sorcerer = Spiritual+Elemental; Empath = Spiritual+Mental. Monk = unarmed combat enhanced by mental magic; Rogue = stealth/ambush/locksmithing; Warrior = heavy-armor frontline. Savant is announced as a future (mental-hybrid) profession. Distinct concept: each profession has a GUILD (profession-only skills, e.g. Rogue lockpick mastery, Bard Enchante, pures' Alchemy) which is separate from cross-profession SOCIETIES (Voln/Sunfist/Light).
  
_Build note:_ Ruleset builder authors a Profession table {name, tier(pure/semi/square), primeStats[2], accessibleCircles[], skillMaxPerLevel matrix (0x/1x/2x/3x per skill), guildSkills[]}. tier drives spell-research max and armor expectations; computed engines never hardcode a class.

**Races (13) with per-stat max bonuses, growth interval, decay timer**  
Thirteen playable races: Aelotoi, Burghal Gnome, Dark Elf, Dwarf, Elf, Erithian, Forest Gnome, Giantman, Half-Elf, Half-Krolvin, Halfling, Human, Sylvankind. Each race has a maximum non-enhanced stat bonus at raw 100 that differs per stat (examples: Giantman Strength +40, Halfling Dexterity +40, Dwarf Constitution +40, Burghal Gnome Logic +35); total bonus budget across all 10 stats ranges ~255-275 per race. Each race also has a Stat Growth Interval (SGI) modifier affecting how fast stats climb on level-up, a body-decay timer used at death (Human ~14 min, Elf ~10 min), and a size that shifts hit-location distribution.
  
_Build note:_ Race table in ruleset {statMaxBonus[10], sgiModifier, decayTimerSec, size}. Worldbuilder can rename/reskin races per authored world; computed max-bonus + growth stays engine-driven.

**Ten statistics, the bonus formula, growth, and what each stat drives**  
Stats: Strength, Constitution, Dexterity, Agility (physical); Aura, Discipline (hybrid); Logic, Intuition, Wisdom, Influence (mental). Raw range 20-100. Bonus = floor((raw-50)/2) + raceModifier (raw 50 => +0 pre-race; +1 per 2 points over 50). Effects: STR->melee AS, encumbrance, stamina, block/parry DS; CON->Health Points, crit/disease resistance, encumbrance; DEX->ranged AS, spell aiming, roundtime reduction; AGI->Defensive Strength; AUR->Elemental CS + mana; WIS->Spiritual CS + spiritual mana; LOG->Mental CS + experience absorption + MTP; INT->experience/mental; DIS->field-exp pool size + mental/emotional resistance; INF->Bard/social CS. Stats grow only at level-up (increase when current level is divisible by floor(currentStat / SGI)); cap 100 raw.
  
_Build note:_ Pure computed function raw->bonus, then a coefficient table wires each stat into AS/DS/CS/TP/HP/exp. All coefficients live in ruleset data so a world can retune stat weightings.

**Character creation constraints and starting points**  
At creation the player distributes 440 attribute points across the 10 stats: minimum 20 in each; only FOUR stats may exceed 70; only ONE may exceed 90; max 100. After allocation the profession grants +10 to each of its two prime stats. Starting training points ~640, +10 per prime => ~660.
  
_Build note:_ Creation wizard validates the 440/20/four-over-70/one-over-90 rules and applies prime +10 from the Profession table; feeds directly into the TP and stat-bonus engines.

**Training points per level (PTP/MTP) and prime doubling**  
Per level: MTP = 25 + floor[(LOG+INT+WIS+INF+((AUR+DIS)/2))/20]; PTP = 25 + floor[(STR+CON+DEX+AGI+((AUR+DIS)/2))/20]. A profession's two prime requisites are DOUBLED inside these formulas. Practical max ~50 PTP / 50 MTP per level, rising to ~60 in a category when both primes fall there (e.g. Rogue with DEX+AGI primes can reach 60 PTP). After level 100 the rate becomes 1 PTP + 1 MTP per 2,500 experience (plus mastery/permanent TP).
  
_Build note:_ Computed engine recomputes PTP/MTP whenever level or a stat changes; prime-doubling reads primeStats from the Profession table.

**Full skill catalog, cost model, diminishing bonus, and rank caps**  
Skills by category — ARMOR: Armor Use (reduces attack RT/spell hindrance/maneuver hindrance; each +20 Armor Use bonus removes 1s of armor RT), Shield Use. WEAPONS: Edged, Blunt, Two-Handed, Polearm, Ranged (bows), Thrown, Brawling; plus Two Weapon Combat, Multi Opponent Combat, Ambush. COMBAT: Combat Maneuvers, Physical Fitness (cost 8/0; gives CMAN defense + health), Dodging. GENERAL: Survival, Disarming Traps, Picking Locks, Stalking and Hiding, Perception, Climbing, Swimming, First Aid, Trading, Pickpocketing. MAGIC: Arcane Symbols, Magic Item Use, Spell Aiming, Harness Power, Mana Control (Elemental/Spiritual/Mental), Spell Research (one per accessible circle), and the Lores — Elemental (Air/Earth/Fire/Water), Spiritual (Blessings/Religion/Summoning), Mental (Divination/Manipulation/Telepathy/Transference/Transformation). Every skill has a PHYSICAL/MENTAL cost (shown as e.g. First Aid '1/2') plus a parenthesized 'times per level'. Cost DOUBLES for each additional purchase within the same level. Bonus per rank has diminishing returns: +5/rank at ranks 1-10 stepping down to +1/rank at ranks 41+. At level 100 skills cap around 101 (1x), 202 (2x), or 303 (3x) ranks depending on how many times/level the profession may train them. (2021 change: CMAN/CML costs are now profession-agnostic.)
  
_Build note:_ THIS is the central ruleset dataset: Skill catalog {name, category, ptpCost, mtpCost}; a Profession x Skill 'maxTimesPerLevel' matrix (0x/1x/2x/3x); the diminishing-bonus bracket table; and skill->derived-stat wiring. Rank->bonus is a pure computed function; the per-level doubling-cost and cap are enforced at the training screen.

**Core combat resolution: AS vs DS + AvD + open d100 endroll**  
EndRoll = AttackStrength(AS) + AvD + d100 - DefensiveStrength(DS); the attack HITS if EndRoll > 100. AvD (Attack-versus-Defense) is a table value from weapon type x defender armor group. The d100 is OPEN-ENDED (~5% chance to roll <1 or >100). AS is built from weapon-skill bonus (scaled by stance) + STR/DEX bonus + spells + enhancives; DS from AGI + Dodging/Shield/Parry (stance-scaled) + spells + enhancives. Example from source: AS +109 vs DS +44 with AvD +39 + d100 +70 = +174 endroll (hit).
  
_Build note:_ The authoritative combat resolver lives on the Evennia server as a pure function reading AvD + DF tables from ruleset data; use a seeded RNG for reproducibility/replay (mirrors the netcode determinism discipline already in the multiplayer spec).

**Damage and critical-rank calculation (armor divisors + randomization)**  
Raw damage = (EndRoll - 100) x Weapon Damage Factor (DF). Critical Rank = truncate[ ((EndRoll-100)xDF - ArmorPaddingPoints + WeaponWeightingPoints) / ArmorCriticalDivisor ]. Armor critical divisors: robes/unarmored 5, leather 6, scale 7, chain 9, plate 11 (so 5-11 raw damage = a rank-1 crit vs unarmored). Critical randomization converts a computed 'max rank' into an actual rank uniformly between ceil(max/2) and max (table: rank1->1, rank2->1-2, rank3->2-3 ... rank9->5-9 max). Fatal thresholds vary by location x damage type (most head/neck/chest/abdomen kills need rank 6-8; head crush/impact fatal at rank 5; eye puncture fatal at rank 4). A crit sets stun duration, wound severity, bleeding, and extra damage.
  
_Build note:_ Critical tables are pure ruleset data keyed [damageType][bodyPart][rank] -> {woundRank, bleedRate, stunSec, extraDamage, fatal}. LLM generates the flavor message string per table cell; the engine only picks the cell.

**Hit locations, wounds/scars, HP, bleeding, death**  
Unaimed hit-location distribution (equal-height): arms 19.7%, legs 16.5%, chest 14.7%, abdomen 12.1%, back 10.1%, hands 9.0%, neck 7.4%, eyes 5.3%, head 5.2%. Wounds have three ranks per body part: R1 minor (mostly combat-maneuver penalties), R2 moderate (deep lacerations/fractures, usually BLEEDS, substantial penalties), R3 severe (severing, blindness, blocks casting/ranged/searching/sneaking depending on location). SCARS are partially-healed remnants (R1 scar = no mechanical penalty; R2 scar penalizes). Bleeding drains Health Points each ~7s round; HP pool scales with CON+level; reaching 0 HP or a lethal head/neck crit = death; the body then decays on the race timer.
  
_Build note:_ Server tick engine advances bleed/HP each round; per-body-part wound state machine feeds injury->penalty modifiers back into the combat and skill engines. Worldbuilder-authored 'you are here' wound diagram is a Godot client panel.

**Healing ecosystem (empath transfer, herbs, first aid, cleric)**  
EMPATH: TRANSFER a patient's wound onto themselves (leaves NO scar), then self-heal via the Empath Base (1100s) circle — able to heal every injury type by roughly level 18; Mental Lore Transference (~to 50 ranks) aids scar transfer, Transformation (~to 57) speeds self-heal; IMPRINT lets an empath remotely transfer wounds/scars (not blood loss) to imprinted patients. HERBS: eating a healing herb reduces a wound one level but leaves a scar; separate scar-removal herbs reduce scarring; herb-eating RT is reduced by First Aid at -1 RT per 20 skill bonus. FIRST AID TEND: bandaging temporarily stops bleeding; bandage duration lengthens and RT shortens with more First Aid. CLERIC/spiritual heals and Raise Dead round it out.
  
_Build note:_ Model healing as data-driven 'effects' (herb table entries, spell effects) that mutate the wound state machine; Empath/Cleric are first-class living-world NPC roles that can heal players using the same engine.

**Magic: three spheres and the spell-circle number map**  
Three spheres: Elemental, Spiritual, Mental. Spell circles and their numeric ranges: Minor Spiritual 100s (shared by ~8 professions), Major Spiritual 200s, Cleric Base 300s, Minor Elemental 400s, Major Elemental 500s, Ranger Base 600s, Sorcerer Base 700s, Wizard Base 900s, Bard Base 1000s, Empath Base 1100s, Minor Mental (~1200s), Paladin Base 1600s, plus Arcane (Sorcerer + shared arcane). A spell number = circle base + slot (e.g. 401, 425, 911). Pures learn 2 minor + their base circle (3 total); semis learn 1 minor + their base; squares up to 2 minor circles. Mana capacity grows via the Harness Power skill; more powerful spells cost more mana.
  
_Build note:_ Spell tables in ruleset {number, circle, sphere, type(warding/bolt/attack/utility/buff/heal), manaCost, castRT, effect, prepMsg}. Worldbuilder + LLM author flavor; computed engine handles cost/effect/hindrance.

**Casting mechanics: prep, cast RT, CS vs TD, bolts, armor hindrance**  
PREPARE is now instantaneous (verb is a legacy hold-over); actual cast roundtime is ~3 seconds; a prepared spell holds ~30s or until released on a valid target. WARDING spells resolve like combat: CS - TD + CvA + d100 > 100 to fully succeed (partial effects at lower margins). Casting Strength (CS) = caster level + Spell Research ranks + casting stat (AUR elemental / WIS spiritual / LOG mental). Target Defense (TD) = target level + defensive magics. CvA is casting-vs-armor. BOLT spells instead use Spell Aiming vs the target's DS (like a ranged attack). Heavier armor causes spell failure (hindrance), reduced by Armor Use.
  
_Build note:_ Two resolver paths on the server: warding (CS/TD/CvA/d100) and bolt (Spell Aiming/DS), both pure functions over ruleset data; armor-hindrance % is a computed value from armor group minus Armor Use.

**Spellburst (anti buff-stacking gate for high-level areas)**  
Spellburst penalizes a character wearing too many BENEFICIAL spells from OUTSIDE their profession's circles when inside flagged high-level hunting areas — inflicting hitpoint loss, critical damage, or even death. A character's safe-buff tolerance is raised by training the magic support skills: Magic Item Use, Arcane Symbols, Harness Power, and Spell Aiming.
  
_Build note:_ Computed 'burst tolerance' = f(MIU, Arcane Symbols, Harness Power, Spell Aiming) vs count/level of foreign active buffs; the server enforces it only in worldbuilder-tagged spellburst rooms.

**Combat Maneuver List (CMAN) — a large data-driven maneuver catalog**  
A character earns 1 CMAN point per rank of Combat Maneuvers; maneuvers are bought from the CML (mastering one typically costs ~20-30 points; since 2021 costs are the same for all professions). Categories and examples — ATTACK: Berserk, Mighty Blow, Coup de Grace, Spin Attack, Staggering Blow, True Strike, Leap Attack, Exsanguinate, Mug. SETUP: Feint (drop stance), Disarm Weapon, Sweep, Trip, Hamstring, Kneebash, Shield Bash, Sunder Shield, Cheapshots, Dirtkick, Eyepoke, Groin Kick, Headbutt, Throatchop, Tackle, Cutthroat, Subdue. PASSIVE/SPEC: Weapon/Parry/Evade/Block/Punch/Kick/Grapple Specialization, Combat Toughness/Mobility/Movement/Focus, Cunning Defense, Precision, Spell Parry. MARTIAL STANCES: Inner Harmony, Predator's Eye, Striking Asp, Whirling Dervish, Rolling Krynch, Duck and Weave, Executioner's Stance, Griffin's Voice, Slippery Mind, Flurry of Blows. BUFFS: Burst of Swiftness, Surge of Strength, Ki Focus, Internal Power, Bearhug, Garrote, Retreat. AoE: Bull Rush, Eviscerate. Shield-specific maneuvers key off Shield Use. Maneuvers consume STAMINA.
  
_Build note:_ CML is a textbook ruleset table {name, category, ranks, pointCost, staminaCost, RT, effect}; add a stamina resource engine. This is the single highest-leverage data-driven system to expose in the ruleset builder.

**Crafting/artisan trades + enchanting + enhancive items**  
Artisan cap 1200 ranks lets a character master 2 artisan skills; Forging is special (its ranks count 1/3 so a forger can master every weapon type). Trades: FORGING (a weapon = 2 parts, a blade/head + a hilt/haft; requires wearing an apron, a self-crafted forging-hammer, and a workshop/forge), ALCHEMY (a guild skill for Cleric/Empath/Sorcerer/Wizard — combine ingredients and process them with equipment to make items/potions), Outfitting/Tailoring, Engineering, Remedies (herbalism), Cobbling (footwear). ENCHANTING via Enchant (925): each success permanently adds +1 to a weapon/armor/shield bonus, paid in ESSENCE (wizards accrue essence = experience; caps 50,000/week and 200,000 total; F2P earns none). Essence cost is tiered (+0 to +24 ~= 312 x bonus level; +25 costs 7,500 then +7,500 per point above 24, e.g. +49->+50 = 195,000). Items must be pre-unlocked with colored ayan'eth (ROYGBIV) potions (green -> +20 in four +5 pours; blue -> +25; higher tiers beyond). Difficulty = trunc[((+Y - 2)^2)/9]. Success factors: Wizard Base ranks (+2/rank up to 1x level), character level, Magic Item Use/Arcane Symbols, LOG/INT, workshop (+25 public / +75 private), familiar. Failure loses 10% of the essence but never destroys the item. ENHANCIVE ITEMS grant stat/skill bonuses with limited charges.
  
_Build note:_ Crafting = recipe tables + a skill-check computed engine; the enchant essence-cost curve and difficulty formula are pure ruleset data; enhancive items are item-property modifiers that feed the stat/skill engines.

**Economy: coins, bank, gem/pawn shops, lockers/vaults, boxes/traps/locks**  
Currency is silver coins (the main unit; copper/bronze subunits exist); large sums convert to promissory NOTES for transport. BANKS in each town deposit/withdraw silvers and notes. GEM/PAWN shops buy gems and loot for silver (collectible shops don't buy). LOCKERS are rented storage — 50 item slots on standard subscription, 100 on premium — plus an account-wide FAMILY VAULT. TREASURE BOXES drop from creatures and are opened via a skill loop: DETECT the box -> if trapped, DISARM it (needs a lockpick) -> PICK it with a lockpick; governed by the Picking Locks and Disarming Traps skills (and Lock Mastery). Mishandling a trap triggers damaging effects.
  
_Build note:_ Coins/items persist in Evennia; bank/shop/locker are worldbuilder-authored room services; the box loop is a computed skill-check minigame (Picking Locks / Disarming Traps bonus vs box difficulty) that makes rogues mechanically distinct.

**Experience: field pool, absorption pulse, and the mind-state ladder (the core loop clock)**  
Earned experience first enters a FIELD (unprocessed) pool of size 800 + LOG + DIS (+ subscription: F2P 0 / Premium 200 / Platinum 400; + Wisdom-of-the-Ages 2/month). It ABSORBS into permanent 'mind' each pulse: on-node = 25 + LOG/5 + poolBonus + groupBonus + superNode(+2); in-town off-node = 22 + LOG/5 + ...; wilderness = 19 + LOG/7 + ...; poolBonus = +1 per 200 unabsorbed (cap +10 at 2000+); grouping with a non-F2P player = +1/pulse. Mind-state ladder by pool fullness: 'clear as a bell' (0%) -> 'fresh and clear' (>0%) -> 'clear' (>25%) -> 'muddled' (>50%) -> 'becoming numbed' (>62%) -> 'numbed' (>75%) -> 'must rest' (>90%) -> 'completely saturated' (>100%, no more field exp banks). Absorbed experience yields training points/levels; level cap is effectively 100, after which you gain mastery/permanent TP per 2,500 exp.
  
_Build note:_ A server 'pulse' scheduler drives absorption and the mind-state gate — this rhythm (hunt to fill the pool, rest/absorb, repeat) IS the GS4 gameplay loop and must be built early. 'Nodes' are worldbuilder-tagged rooms with an absorption multiplier.

**Death, deeds, resurrection, societies, subscription tiers**  
On death, unabsorbed field experience is LOST unless a Chrism is used; the corpse decays on the race timer; the character suffers Death's Sting (Constitution reduction, sharply reduced absorption). DEEDS are small purchased favors from Lorminstra (goddess of death/rebirth); one is auto-consumed per death to cushion the penalty. Resurrection: Cleric Raise Dead (318), Symbol/shrine revival. SOCIETIES are cross-profession and you may belong to only one at a time: Order of Voln (banishing undead; one new symbol per rank, 26 at Master), Guardians of Sunfist (anti-Grimswarm; HQ in Zul Logoth), Council of Light. Guilds (profession-specific) are separate from societies. Subscription tiers (F2P / Premium / Platinum) scale the exp pool, locker size, essence, and estate/estate-holder perks.
  
_Build note:_ Society membership = an unlockable ability track (rank -> granted symbol/ability), pure ruleset data; deeds are a death-insurance micro-economy; worldbuilder authors society HQs, shrines, and guild halls.

**Living world / autonomous agents (our differentiator over GS4)**  
GS4's world is persistent and hand-authored but its NPCs are largely static scripts. Our edge is simulated agent lives: town NPCs, shopkeepers, roaming adventurers, and society members with real personalities, psychology, daily schedules, relationships, and reactions to world events (e.g. a Grimswarm invasion, player reputation, festivals). Because the medium is text, the simulation can be arbitrarily deep.
  
_Build note:_ An NPC = a WORLDBUILDER entity (home/job/relationships/timeline) + a COMPANION-AI personality/psychology (consume-only) + an LLM voice (dialogue/rumors) + an Evennia daily-routine scheduler. Critically, NPCs are first-class actors that use the SAME computed combat/skill/magic engines as players, so an autonomous ranger hunts and an empath heals by the identical rules.

### How our apps feed it

- RULESET builder = the system-of-record for all GS4 numeric tables: Profession defs (tier pure/semi/square, prime stats, accessible spell circles, per-skill max-times-per-level matrix, guild skills), Race defs (per-stat max bonus, Stat Growth Interval, decay timer, size), Skill catalog (physical/mental costs, diminishing +5..+1 brackets, caps), Spell circle/spell defs (number, sphere, type, mana, cast RT, effect), AvD tables, Weapon Damage Factors, Critical tables keyed [damageType x bodyPart x rank], CMAN/CML maneuver catalog, Enchant essence curve + difficulty, Society ability tracks, and the experience-pool/absorption/deed formulas. Authoring here reskins GS4 into any world with zero engine code changes.
- Computed engines (pure functions over ruleset data) = the runtime evaluators: stat->bonus, PTP/MTP-per-level with prime doubling, stat growth, rank->skill-bonus, AS/DS/endroll, damage & critical rank, warding CS-TD and bolt Spell-Aiming resolution, mana capacity, absorption-per-pulse, wound/bleed/HP ticks, roundtime, box/lock skill checks, enchant success. Run authoritatively on the Evennia server; identical for players and NPCs.
- WORLDBUILDER = authors Elanthia itself: rooms + descriptions, town maps, experience 'node' tags, shops/banks/gem-pawn/lockers as room services, society HQs + guild halls + shrines, creature spawns + encounter/loot tables, spellburst-area flags, and the lore/timeline/faction state (e.g. Grimswarm invasions) the living world reacts to.
- COMPANION AI library (consume-only) = gives autonomous NPCs their inner lives — personality/psychology/relationships/schedules — so shopkeepers, townsfolk, roaming adventurers, and society members behave like simulated people rather than static scripts. Never edited; integrated read-only.
- LLM generation = surface text at scale: room descriptions, NPC dialogue and rumors, dynamic quest/flavor text, and the per-cell critical/wound message strings that the deterministic engine selects (engine picks the table cell; LLM voices it).
- Evennia (BSD) server = authoritative persistent simulation: combat/magic/skill resolution, roundtime + experience pulses + world tick, NPC daily-routine scheduling, and all coin/item/character persistence. Godot client = the GS4-parity text UI/UX (command line + narrative pane + status/health/mind/RT/wound panels).
- Existing netcode/multiplayer drop-in = the host-authoritative + seeded-determinism discipline (from MULTIPLAYER_TEMPLATE_SPEC) carries over, but an MMORPG uses the persistent Evennia server as the single source of truth rather than the peer-host 'authority-turn' model; the SessionState-style 'clients render off server signals, never lead' pattern is the right client contract.
